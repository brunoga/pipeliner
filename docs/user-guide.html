Pipeliner User Guide

Pipeliner is a media-automation daemon that pulls entries from RSS feeds, torrent indexers, or local filesystems, enriches them with metadata, applies configurable filters, and hands accepted entries off to download clients, notification services, or shell commands.

It is heavily inspired by FlexGet. Pipelines are described in Starlark — a deterministic, side-effect-free dialect of Python used by Bazel and Buck. Each task chains a sequence of plugins. The scheduler runs tasks on cron or interval schedules, persisting state in a SQLite database so each entry is processed only once.

Key concepts

  • Entry — a title, URL, state (undecided/accepted/rejected/failed), and a key-value field map
  • Plugin — a self-contained unit that produces, annotates, filters, modifies, or acts on entries
  • Task — an ordered chain of plugins; metainfo, filter, and modify plugins run in config-file order
  • Phase — the execution category a plugin belongs to (input, metainfo, filter, modify, output, learn)

Quick Start

Create config.star:

task("breaking-bad",
    [
        plugin("rss", url="https://example.com/rss"),
        plugin("seen"),
        plugin("series", static=["Breaking Bad"]),
        plugin("transmission", host="localhost", port=9091),
    ],
    schedule="1h")

Run once to test:

pipeliner run --config config.star

Run as a daemon (with web UI):

pipeliner daemon --config config.star --web :8080 \
  --web-user admin --web-password secret

Validate config without running:

pipeliner check --config config.star

Configuration File

Pipeliner uses a single Starlark script (default: config.star). Pass a custom path with --config. Starlark is a deterministic, side-effect-free Python dialect — if you know Python, you already know Starlark.

Built-in functions

Three functions are available globally in every config script:

FunctionDescription
plugin(name, key=val, …)Create a plugin config using keyword arguments
plugin(name, {…})Create a plugin config using a dict (required when config keys conflict with Starlark identifiers)
task(name, plugins, schedule="", priority=0)Register a named task; plugins is a flat list of plugin() calls
env(name, default=None)Read an environment variable; errors if unset and no default given
load("./other.star", "sym")Import a symbol from another .star file (Starlark built-in)

Structure overview

# Variables — ordinary Starlark assignments
tv_root  = "/media/tv"
tvdb_key = env("TVDB_API_KEY")   # from environment

# Templates — Starlark functions returning lists of plugin() calls
def common_input(url):
    return [plugin("rss", url=url), plugin("seen")]

# Tasks — call task() once per pipeline
task("my-task",
    common_input("https://example.com/rss") + [
        plugin("series", static=["My Show"]),
    ],
    schedule="1h")

Database

Pipeliner automatically maintains pipeliner.db (SQLite) in the same directory as the config file. All stateful plugins (seen, series, movies, premiere, upgrade, the TVDB/Trakt caches, and persistent lists) share this file. No configuration needed.

Variables

Variables are ordinary Starlark assignments at the top of the script. Use env(name) to read from the process environment. Both produce regular Starlark strings usable anywhere.

PatternDescriptionExample
name = "value"Inline literaltv_root = "/media/tv"
name = env("VAR")Required env var — errors if unsetapi_key = env("TVDB_API_KEY")
name = env("VAR", default="x")Optional env var with fallbackhost = env("DELUGE_HOST", default="localhost")
tv_root  = "/media/tv"
tvdb_key = env("TVDB_API_KEY")   # must be set in the process environment

task("tv", [
    plugin("metainfo_tvdb", api_key=tvdb_key),
    plugin("pathfmt", path=tv_root + "/{title}", field="download_path"),
])

Because variables are plain Starlark values you can also compute them at config load time using string concatenation, conditionals, or any other Starlark expression. There is no special substitution syntax — the language itself handles composition.

Templates

Templates are ordinary Starlark functions that return a list of plugin() calls. Because they are real functions, they support any parameter type (strings, numbers, lists, dicts), default values, conditional logic, and loops — without any special syntax.

# Template with typed parameters and a default
def common_input(url, local_seen=False):
    return [
        plugin("rss", url=url),
        plugin("seen", local=local_seen),
    ]

def common_output(dest, host="localhost", port=9091):
    return [
        plugin("pathfmt", path=dest + "/{title}", field="download_path"),
        plugin("transmission", host=host, port=port, path="{download_path}"),
    ]

def jackett_search(indexers, categories):
    return [plugin("jackett_input",
                   indexers=indexers, categories=categories)]

def email_notify(subject, body_template):
    return [plugin("email",
                   subject=subject, body_template=body_template,
                   smtp_host=smtp_host, smtp_port=smtp_port,
                   from_=smtp_user, to=smtp_to)]

# Using templates — just call them like functions
task("tv-shows",
    common_input("https://example.com/rss") +
    jackett_search(["torrenting", "showrss"], ["5000"]) +
    [plugin("series", static=["My Show"])] +
    common_output("/media/tv") +
    email_notify(
        subject="New episodes: {{len .Entries}}",
        body_template="{{range .Entries}}<p>{{index .Fields \"title\"}}</p>{{end}}",
    ))

Key points

  • Templates return a list of plugin() values. Concatenate lists with + to compose them into a task.
  • Parameters accept any Starlark type: strings, numbers, lists, dicts, or other functions. No special wrapping needed.
  • Default parameter values work as in Python: def f(x, y="default"):
  • Use load("./common.star", "my_template") to share templates across config files.
  • Starlark is deterministic — no side effects, no imports from the network, no file I/O — so configs are safe to evaluate repeatedly.

For the full Starlark language reference see the Starlark specification and the starlark-go implementation used by pipeliner.

Schedules

In daemon mode, tasks run automatically according to their schedule. Pass the schedule as the schedule= keyword argument to task(). Both interval strings and 5-field cron expressions are supported.

task("hourly-task",  [...], schedule="1h")
task("daily-task",   [...], schedule="24h")
task("morning-task", [...], schedule="0 7 * * *")   # every day at 07:00

Tasks without a schedule= argument are not run automatically; trigger them manually via the web UI or pipeliner run task-name.

Pipeline Phases

All input plugins run first (concurrently), then metainfo, filter, and modify plugins run in the order they appear in the config, and finally all output plugins run (concurrently). The learn phase follows output.

Input

Metainfo
↕
Filter
↕
Modify
→
Output
→
Learn
PhaseWhat happensEntry state after
InputPlugins produce entries from sources (RSS, files, searches). All inputs run concurrently; results are merged and deduplicated by URL.undecided
MetainfoPlugins annotate entries with additional fields (quality tags, TVDB data, torrent info). Runs in config order, interleaved with filter and modify.undecided
FilterPlugins accept, reject, or leave entries undecided. Runs in config order, interleaved with metainfo and modify.accepted / rejected / undecided
ModifyPlugins mutate accepted entry fields (path formatting, field setting). Runs in config order, interleaved with metainfo and filter.accepted (unchanged)
OutputPlugins act on accepted entries in config order. Each plugin receives only entries still accepted at that point — a plugin that fails an entry removes it from subsequent output plugins and from the learn phase.accepted / failed (if downloader errors)
LearnPlugins persist state for future runs (mark as seen, record series tracker). Receives only accepted entries — output-failed entries are excluded and will be retried. Skipped in dry-run mode.accepted only
Metainfo, filter, and modify plugins run in config-file order. Place a filter immediately after the metainfo plugin that sets the field it inspects to reject cheap early and avoid unnecessary annotation work for entries that will be rejected anyway.

Entries & Fields

Each entry carries: title, URL, state (undecided / accepted / rejected / failed), and a string-keyed field map that plugins read and write.

Quality Specs

Used in series, movies, premiere, and the quality filter. A spec is one or more space-separated tokens, each constraining one quality dimension.

FormMeaningExample
valueExact value1080p, bluray
value+This value or better720p+, webrip+
min-maxInclusive range720p-1080p

Recognized dimension values (low → high)

DimensionValues
Resolutionsd, 480p, 576p, 720p, 1080p, 2160p/4k
Sourcedvdrip, tvrip, hdtv, webrip, webdl/web-dl, bluray, remux
Codecxvid, divx, h264/x264, h265/x265/hevc, av1
Audiomp3, aac, dd/dolbydigital, dts, truehd, atmos
Color rangesdr, hdr, hdr10, dv/dolbyvision
3D format3dconv/conv, 3d/3d-half/half, 3d-full/full/sbs/ou, bd3d/bd

Example: 720p+ webrip+ requires at least 720p resolution AND at least WEBRip source quality.

When a 3D token is included, non-3D entries are rejected automatically. Omitting a 3D token leaves the dimension unconstrained — both 3D and non-3D entries pass. Examples: 1080p+ 3d+ (any 3D), 1080p+ bd3d (BD3D only), 1080p+ 3dconv+ (converted 3D or better).

3D quality

3D format is a ranked dimension used only for movie comparisons. When two 3D copies of the same movie are compared, 3D format takes precedence over all other dimensions (resolution, source, etc.), which then act as tie-breakers. 3D and non-3D versions of the same movie are tracked independently — both are downloaded if both match.

RankFormatSpec tokensDetected markers
Lowest3D-Conv3dconv, conv3DCONV, 3D-CONV — artificially converted from a 2D source; always wins over any other format marker in the same title
LowHalf3d, 3d-half, half3D, HSBS, H-SBS, HALF-SBS, HalfSBS, HOU, H-OU, HALF-OU, HalfOU
HighFull3d-full, full, sbs, ouSBS, FSBS, F-SBS, FULL-SBS, FullSBS, OU, FOU, F-OU, FULL-OU, FullOU
HighestBD3Dbd3d, bdBD3D, MVC codec, or any native 3D marker combined with COMPLETE BluRay (full disc rip)

When a title contains multiple 3D markers (e.g. IMAX 3D FSBS), the highest-ranked one wins — the explicit format tag beats the generic 3D label. If 3DCONV is present anywhere in the title, the result is always 3D-Conv regardless of other markers.

Use video_is_3d to test for any 3D release. The 3D format appears in video_quality (e.g. BD3D 1080p BluRay H.265).

Data migration note: The 3D-Conv tier was inserted at integer value 1, shifting the stored values of all existing tiers upward (Half: 1→2, Full: 2→3, BD: 3→4). If your tracker database contains 3D movie entries from before this change, their stored format integers will be misread. Clear or re-track any affected 3D movies via the Database → Movies tab in the web UI, or by deleting the relevant rows from pipeliner.db.

Input Plugins

Input plugins produce entries. All inputs in a task run concurrently; their results are merged and deduplicated by URL before other phases begin.

discover

Actively searches multiple sources for entries matching a title list. Unlike RSS (which passively receives all items), discover iterates titles, dispatches one search query per title to each configured search plugin (via), and returns the merged results. From Plugins

From plugins are sub-plugins — they cannot be used as top-level task plugins and will cause a validation error if you try. They are referenced inside series.from, movies.from, discover.from, or discover.via.

  from:
    - name: tvdb_favorites   # from plugin with inline config
      api_key: YOUR_KEY
      user_pin: YOUR_PIN
    - trakt_list            # from plugin by name alone (uses defaults)

Fetches a user's TheTVDB favorites list and emits one entry per show. Use inside series.from or discover.from to build a dynamic show list from your TVDB account.

Trakt Authentication

Private Trakt lists (watchlist, ratings, collection) require an OAuth token. Pipeliner supports two approaches:

Recommended: managed tokens via client_secret

Complete the device auth flow once to obtain and store a token in pipeliner.db. Pipeliner then refreshes it automatically before expiry (tokens last 90 days). There are two ways to do this:

Via the web UI (recommended for Docker) — open the Settings tab, enter your client_id and client_secret, click Authorize, then visit the displayed URL and enter the code on Trakt's website. The token is saved automatically.

Via the CLI — run the auth command, then visit the displayed URL and enter the code:

  1. Create a Trakt application at trakt.tv/oauth/applications to get a client_id and client_secret.
  2. Run the auth command (use the same --config path as your pipeliner config so the token is stored in the right pipeliner.db):
    pipeliner auth trakt --client-id=YOUR_ID --client-secret=YOUR_SECRET --config=config.star
  3. Visit the URL shown and enter the displayed code on Trakt's website.
  4. Use client_secret in your config — no access_token needed:
    from:
  - name: trakt_list
    client_id: YOUR_CLIENT_ID
    client_secret: YOUR_CLIENT_SECRET
    type: movies
    list: watchlist

Alternative: static access_token

Obtain a token manually from Trakt and place it directly in the config. Tokens expire after 90 days and must be updated manually.

from:
  - name: trakt_list
    client_id: YOUR_CLIENT_ID
    access_token: YOUR_TOKEN
    type: movies
    list: watchlist

Filter Plugins

Filter plugins run in config order, interleaved with metainfo and modify plugins. Each entry is passed to every filter in sequence; once an entry is rejected or failed it is skipped by subsequent filter plugins.

series

Accepts episodes of configured TV shows. Parses the episode identifier from the title, matches series names with fuzzy matching, and enforces optional quality and ordering constraints. Multiple quality variants of the same episode are all accepted — the task engine's automatic deduplication picks the best copy. Download history is recorded in the Learn phase. A re-download of an already-seen episode is accepted when the new copy is strictly better quality, or when it is a PROPER/REPACK that is not a quality downgrade.

KeyRequiredDefaultDescription
staticconditional—Static list of show names
fromconditional—From-plugin configs providing show names dynamically
ttlno1hCache TTL for dynamic show list. An empty list from a from: source is not cached — the source will be queried again on the next run.
trackingnostrictEpisode ordering mode: strict, backfill, or follow
qualityno—Minimum quality spec (e.g. 720p+ webrip+)
reject_unmatchednotrueWhen true (default), entries that do not parse as a series episode or whose show name is not in the configured list are explicitly rejected. Set to false to leave such entries undecided — useful when chaining multiple series filters in one pipeline.

Tracking modes

ModeBehaviour
strictAccept only the next expected episode; reject gaps >1 ahead of latest downloaded. Does not handle full-season binge drops well.
backfillAccept any episode not yet downloaded, including older ones. Downloads full back-catalogue.
followRecommended for new and continuing series. On first encounter accept everything (full season drops in one pass). Afterwards the earliest tracked season acts as an anchor — episodes from older seasons are rejected, all episodes in the anchor season or newer are accepted (including gap-fills within the current season).

Fields set: series_season, series_episode, series_episode_id. The matched show name is written to title (canonical) and raw_title (original parsed name).

series:
  tracking: strict
  quality: 720p+ webrip+
  static:
    - "Breaking Bad"
    - "Better Call Saul"
# Dynamic list from TVDB favorites — backfill all episodes not yet downloaded
series:
  tracking: backfill
  quality: 720p+ webrip+
  from:
    - name: tvdb_favorites
      api_key: YOUR_KEY
      user_pin: YOUR_PIN

KeyRequiredDefaultDescription
staticconditional—Static list of movie titles
fromconditional—From-plugin configs providing titles dynamically
ttlno1hCache TTL for dynamic list. An empty list from a from: source is not cached — the source will be queried again on the next run.
qualityno—Minimum quality floor
reject_unmatchednotrueWhen true (default), entries whose title does not match any title in the configured list are explicitly rejected. Set to false to leave such entries undecided — useful when chaining multiple movies filters in one pipeline.

Fields set: title (movie title), video_year, video_quality, video_resolution, video_source, video_is_3d

3D handling: 3D and non-3D versions of the same movie are tracked independently — both are downloaded if both match. Among competing 3D copies, 3D format quality (BD3D > Full SBS/OU > Half SBS/OU) takes precedence over resolution and source. Use condition: reject: 'video_is_3d == true' to exclude all 3D releases.

Rejects torrents with fewer than a minimum number of seeds. The torrent_seeds field is set by the RSS input, Jackett, and jackett_input. Entries without seed data are left undecided.

KeyRequiredDefaultDescription
min_seedsno1Minimum seed count required
torrent_alive:
  min_seeds: 3

require

Rejects entries missing one or more required fields. A field is "missing" if its value is nil, empty string, zero, false, or empty slice.

require:
  fields:
    - enriched
    - series_episode_id
Using enriched as a metainfo gate. External metainfo plugins (TVDB, TMDb, Trakt) set enriched = true only when they successfully identify the entry in the database. Adding enriched to require rejects anything the provider did not recognise — for example, a foreign-language title, a very new release not yet in the database, or a torrent that doesn't match any known show or movie. This is the provider-agnostic replacement for the old pattern of requiring tvdb_id: 
metainfo_tvdb:
  api_key: "…"
require:
  fields: ["enriched"]   # drop anything TVDB didn't find
Because enriched is set by any external provider, the same config works unchanged if you later swap TVDB for TMDb or Trakt.

seen

Rejects entries already processed in a previous run, using a SHA-256 fingerprint of configured fields stored in pipeliner.db. Fingerprints are written in the Learn phase — if an output plugin fails, the entry is not marked seen and will be retried.

Not needed for TV series, movies, or premieres. Those filters maintain their own download trackers and handle deduplication automatically. Use seen for pipelines without a domain-specific tracker — the most common case is RSS news or article feeds, where the same article would otherwise be re-processed every run.
KeyRequiredDefaultDescription
fieldsno["url"]Fields to include in the fingerprint
localnofalseScope seen store to this task name only

Typical use — tech news to email, preventing re-delivery of already-sent articles:

tasks:
  tech-news:
    rss:
      url: "https://feeds.arstechnica.com/arstechnica/technology-lab"
    seen:                  # reject already-sent articles by URL
    regexp:
      accept: "(?i)linux|golang|open.?source"
    email:
      smtp_host: smtp.gmail.com
      subject: "{{len .Entries}} new article(s)"

schedules:
  tech-news: 1h

content

Rejects or requires entries based on filenames. The file list is resolved in order:

  1. torrent_files — set by metainfo_torrent or metainfo_magnet (DHT). Full file list from the torrent.
  2. file_location basename — set by the filesystem plugin for local files.
  3. URL path component — last segment of the entry URL, for direct-download entries where the URL reveals the file type (e.g. http://tracker.com/file.rar).

When none of the above is available the check is skipped and a Warn is logged. Run with --log-level debug to see which source was used.

Patterns use path.Match semantics, tested against both the filename component and the full path.

content:
  reject:
    - "*.exe"
    - "*.bat"
  require:
    - "*.mkv"

Modify Plugins

pathfmt

Renders a path template into a named entry field, automatically scrubbing each path component so the result is safe to use as a filesystem path. Characters invalid on common filesystems (<>:"/\|?* and control characters) are replaced with underscores.

KeyRequiredDescription
pathyesTemplate string (supports {field}, {field:format}, and {{.field}})
fieldyesEntry field to write the rendered path into (e.g. download_path)
pathfmt:
  path: "/media/tv/{title}/Season {series_season:02d}"
  field: download_path

A colon in a show title like "Breaking Bad: Season 1" becomes an underscore, yielding Breaking Bad_ Season 1. No separate pathscrub step is needed.

set

Unconditionally sets one or more entry fields. Each config key becomes a field name; its value is a Go template rendered against the entry.

set:
  category: tv
  label: "{{.title}}"

Output Plugins

Output plugins act on accepted entries after modify. Outputs run serially in config order. Each plugin receives only the entries still accepted at the time it runs — a plugin that calls e.Fail() removes that entry from subsequent output plugins and from the learn phase, so it will be retried on the next run.

Error handling: Downloader plugins (deluge, qbittorrent, transmission, download) mark an entry as failed when they cannot queue or fetch it. Failed entries are skipped by the Learn phase and will be retried on the next run. Notification plugins (email, exec, print, notify) do not mark entries failed — a notification failure does not cause a retry.

Web UI

Start the web UI with pipeliner daemon --web :8080 --web-user admin --web-password secret. Navigate to http://localhost:8080.

The interface has four tabs: Dashboard, Config, Database, and Settings. A User Guide link in the header opens this document.

Dashboard tab

  • Task cards — each card shows the next scheduled run datetime (hover for the raw cron/interval), the last run time, accepted/rejected/failed counts, and a Run now button.
  • Run all — triggers all tasks immediately.
  • Reload config — reloads config.star from disk without restarting the process. Queued for after running tasks finish if any are active.
  • Live Log — SSE-streamed log with a filter box. Reconnects automatically; replays the last 10,000 lines for new connections.

Config tab

A live Starlark editor with syntax highlighting. Use Validate to check config without saving, or Save & Reload to write and apply immediately (or queue a reload if tasks are running). Tab key inserts two spaces.

Database tab

Inspect and manage the two download trackers stored in pipeliner.db.

A sidebar lists the available trackers; click one to load its contents in the main panel. A filter box and Clear all button appear for the active tracker.

  • Series — all TV episodes recorded as downloaded, grouped by show. Expand a show to see each episode's ID, quality, and download date. Delete an individual episode or an entire show to force a re-download on the next run. Includes entries from both the series and premiere filters.
  • Movies — all movies recorded as downloaded. Delete a movie to force a re-download.
  • Seen — entries recorded by the seen filter, showing title, task name, and date. Delete an entry to allow it to be processed again. Per-task local seen stores (local: true) appear as separate sidebar entries named Seen: task-name.

Settings tab

The Settings tab provides a Trakt Authorization form for completing the Trakt device auth flow directly in the browser — useful when running in Docker or any environment where interactive console access is not available. Enter your client_id and client_secret, click Authorize, then visit the displayed URL and enter the code on Trakt's website. The token is saved to pipeliner.db automatically on success and refreshed before expiry, exactly as if you had run pipeliner auth trakt from the CLI.

HTTPS

Pass --tls-cert and --tls-key for a real certificate, or --tls-self-signed to generate one automatically. With TLS the Secure cookie flag is set automatically.

Dry-Run Mode

Pass --dry-run to pipeliner run to execute the full pipeline (input → processing → output skipped) but skip the output and learn phases. The run is fully idempotent — no torrents are queued, no emails sent, no state written to pipeliner.db. You can run it repeatedly to inspect what would be accepted without side effects.

pipeliner run --config config.star --dry-run --log-level debug

Troubleshooting

No entries accepted

  • Run with --log-level debug to see why each entry is rejected. The reject reason is logged for every rejected entry.
  • Add print: before output plugins to confirm entries reach the output phase.
  • Check --dry-run output — if entries are accepted in dry-run but not live, the output plugin may be failing.

TVDB not finding a series

  • The series name parser is strict. Add --log-level debug --log-plugin metainfo_tvdb to see what name it searches for. Combine plugins with a comma: --log-plugin metainfo_tvdb,series.
  • The TVDB search uses the parsed series name — if the release title uses an unusual name variant, TVDB may return the wrong result or nothing.
  • Check that your API key and user PIN are correct at thetvdb.com/api-information.

Deluge "Not authenticated" errors

This was a bug where the Deluge session cookie from login was not preserved. It is fixed in v0.1.3+. Ensure you are running a current version.

torrent_alive is slow

When entries lack a torrent_seeds field, torrent_alive must resolve the info hash and scrape trackers. Use inputs that provide seed counts (Jackett, jackett_input, nyaa RSS) to avoid scraping. Set scrape: false to disable tracker scraping entirely (entries without seed data will be left undecided).

Config validation errors

pipeliner check validates each plugin against its fully expanded config, so required fields supplied inside Starlark functions are correctly recognized. If you see "required field missing" for a field your function provides, check that the function is actually called and its return value is included in the task's plugin list.

Schedule not running

Tasks without a schedule= argument in their task() call are not run automatically by the daemon. Add a schedule or trigger manually with pipeliner run task-name or the web UI's Run now button.

PhaseFrom plugin used as top-level input

If you see "plugin X is a from plugin; use it via series.from, movies.from, or discover.from instead", move the plugin inside a "from" list in the parent plugin's config dict:

# Wrong — top-level task plugin:
task("t", [plugin("tvdb_favorites", api_key="YOUR_KEY")])

# Correct — inside series "from" list:
task("t", [
    plugin("series", {
        "from": [{"name": "tvdb_favorites",
                  "api_key": "YOUR_KEY", "user_pin": "YOUR_PIN"}],
    }),
])
