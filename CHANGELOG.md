## [1.10.0] - 2026-06-09

### Added

- **File-backed log viewer with predictable filtering** ([#267](https://github.com/brunoga/pipeliner/pull/267)). The dashboard's Live Log used to feel unpredictable in three concrete ways: a client-side `MAX_LINES = 500` cap silently evicted DOM nodes (lines that scrolled past were gone unless you reloaded), the filter only toggled `display:none` on whatever happened to be in memory (so the same query returned different matches depending on how far you'd scrolled), and SSE warm-up + a separate `/api/logs/history` endpoint paginated independently with overlap-trim bookkeeping on top of "lines-from-EOF" cursors that shifted under each new append. The motivating user complaint was the second one — searching for `"transmission"` after a fresh page-load found the running task's matches, but searching for the same string after scrolling up half a screen found three more entirely different ones, because what the filter "could see" depended on scroll position. The fix is a structural rewrite. A new `internal/web/logfiles.go` package walks the rotating set (`pipeliner.log` + `.1..MaxArchives`) with byte-stable `LinePos{FileIdx, ByteEnd}` cursors and case-insensitive whitespace-AND substring filtering via `Tail`/`Before`/`After` methods, all of which keep scanning past non-matches so sparse filters still fill a page in one call. Three new endpoints expose this — `/api/logs/tail` returns the newest matching N (with `older_cursor` and `exhausted`), `/before` pages strictly older than a cursor, `/after` strictly newer (with `at_tail` when caught up). `/api/logs/history` is removed. The SSE channel at `/api/logs` switches its wire format to JSON `{pos, text}` payloads, accepts `?q=` for server-side filtering, and surfaces an `event: rotate` frame when the rotating writer rolls over mid-session. The rotating writer (`internal/logfile/rotating.go`) gains `OnLine(text, byteEnd)` and `OnRotate()` callbacks fired under its own lock after every successful append/rotation; the `Broadcaster` (`internal/web/logstream.go`) is no longer an `io.Writer` but a `Publish(text, byteEnd)` pub-sub fed directly by the file writer's hooks, so every SSE event carries the stable on-disk cursor of its line. The colored slog leg that previously fed the broadcaster is gone — live and historical lines both arrive plain from the file and the client's existing `plainLogToHtml` colourises them identically, so live-tail and scrolled-back lines now look the same instead of one being ANSI-coloured and the other not. Per-run buffer wipe (`bcast.Reset()` on idle→running) is removed because history shouldn't be wiped between runs — that's the whole point of file-backed history. The client (`internal/web/ui/dashboard.js`) is a single sliding-window state machine over `(topCursor, bottomCursor)` with a window cap of ~2000 rendered lines: scrolling within `LOG_SCROLL_EDGE_PX` of the top fetches `/before`, scrolling within edge of the bottom (when not at live tail) fetches `/after`, the window evicts from whichever end the user is scrolling *away from* and re-anchors the cursor so re-scroll re-fetches correctly. Scrolling off the bottom pauses auto-scroll and surfaces a `"↓ N new"` pill (anchored bottom-right of the console via a new `.log-console-wrap` positioning container); clicking the pill or scrolling back to the bottom re-engages live-follow. The filter is debounced ~150 ms; changes tear down state, refetch `/tail?q=`, and reopen SSE with the same `q` so live matches keep streaming under the active filter. A new SSE reconnect bridge handles the case where a disconnect lasts longer than the broadcaster's 256-line ring can replay: when an incoming live line's byte position implies a gap from `lastLivePos`, the client queues it, fetches `/api/logs/after` to fill the missed window, deduplicates by position, then resumes — so the rendered viewport stays contiguous across long disconnects without losing lines from view. Test coverage is the same shape the rest of the codebase uses: `internal/web/logfiles_test.go` exercises Tail/Before/After across files and filters, cursor monotonicity, a long line straddling > 8 KiB chunk boundaries, the sparse-filter "keep scanning past non-matches" case, and empty/missing-file edge cases; `internal/web/server_test.go` covers the new endpoints' wire shape (including bad-cursor 400, limit clamp, no-log-file-configured behaviour), SSE server-side `?q=` filtering, and end-to-end `event: rotate` delivery to live subscribers; `internal/logfile/rotating_test.go` covers the new callbacks (offset monotonicity, multi-line writes, rotation firing); `internal/web/ui/tests/log-history.test.js` covers boot via `/tail`, scroll-up `/before` paging, SSE live-follow vs paused with the pill counter, `resumeLiveTail` iterating through multiple `/after` pages until `at_tail`, filter teardown reboot, `handleRotation` re-tail, window-cap eviction re-anchoring `topCursor`, and both the reconnect-bridge fire and the no-bridge-when-contiguous fast path. `docs/user-guide.html`'s Live Log description is rewritten to reflect the new behaviour (cross-run scrollback, file-wide search, pill, no in-memory eviction).

### Removed

- **`GET /api/logs/history`** ([#267](https://github.com/brunoga/pipeliner/pull/267)). Subsumed by `/api/logs/tail` + `/before` + `/after`. The endpoint was an internal contract between the daemon and the bundled SPA; both are updated in lockstep, so this isn't a user-facing breaking change. The 1.x.y `Broadcaster.Reset()` per-task-run log wipe is also gone (no longer meaningful when the live tail is backed by the file).

**Why 1.10.0**: substantial new user-perceivable feature (the dashboard log viewer now behaves like a proper log tail with searchable history across runs) plus a sizeable internal refactor (file-as-source-of-truth, new pub-sub broadcaster, cursor-based pagination, server-side filter). Comparable in scope to v1.8.0 (swap_state + InputStates) and v1.9.0 (per-state DAG analysis), both of which were minor bumps. No config or plugin contract changes; existing pipelines unchanged.

## [1.9.2] - 2026-06-08

### Changed

- **Visual editor palette: drop the leading role-color dot, widen the column 200 → 220px** ([#264](https://github.com/brunoga/pipeliner/pull/264)). The forcing function was the `bluray_releases` chip overflow (see Fixed), but the same audit pass surfaced that the small role-color dot at the start of every chip was redundant with the section header above each group ("Sources" / "Processors" / "Sinks") — the dot was decorative reinforcement of information the user just read three pixels above. Removing it reclaims ~11 px of horizontal room across every chip (5 px dot + 6 px gap) and lets the palette feel a touch denser without losing any information. Chip horizontal padding tightened from `4px 8px` → `4px 6px` for the same reason. The palette grid column widened from 200 px → 220 px to give breathing room for worst-case long names with both capability badges (`bluray_releases search list`), so this still fits without truncation after the tightening pass.
- **`bluray_releases` README and user-guide call out the firehose semantics of `.list`** ([#265](https://github.com/brunoga/pipeliner/pull/265)). `bluray_releases` is the only stock list source that isn't user-curated — `trakt_list` reads a Trakt list you maintain, a literal `["Inception", …]` is a hand-picked array, but `bluray_releases.list` is *every* release in the configured window (typically 100+ titles per month). A new user could reasonably wire it like `trakt_list` and discover the firehose by accident. The README and `docs/user-guide.html` `bluray_releases` section both gain a worked example showing the canonical use case (use the calendar as a *curation proxy* against a noisy RSS torrent feed rather than as the entry source itself) and the inverse guidance: if you find yourself stacking many downstream `condition` rules to suppress noise, the firehose model isn't right — switch to `trakt_list` or a literal title array so the list reflects intent, not a calendar. Example config validates cleanly under `pipeliner check`.

### Fixed

- **Dashboard "Run now" hover colour decouples from the card stripe** ([#263](https://github.com/brunoga/pipeliner/pull/263)). `.btn-run:hover` used `var(--card-color, var(--accent))`, with `--card-color` set inline on every task card by `dashboard.js` based on last-run health (running → accent, failed → red, rejected → amber, accepted → green, **no history → border-grey**). Because the property is always set, the `var(--accent)` fallback never fired — and pipelines with no run history (typical for manual-trigger pipelines without a schedule) hovered to muted grey instead of an action colour. The coupling was also semantically noisy on cards with history: a red card hovered red (reads destructive), an amber card hovered amber (visually conflated with the Dry button's amber). Hover now uses `--accent` directly, matching `.btn-run.running` so hover → click → running is one continuous colour; the side stripe keeps its job of encoding last-run health at a glance. `.btn-run.btn-dry:hover` is unchanged (already fixed amber). The regression test inspects the `.btn-run:hover` rule's `borderColor` source string directly because headless Chromium does not reliably trigger CSS `:hover` via synthetic mouse moves.
- **Both capability badges on long-named palette chips are now actually visible** ([#264](https://github.com/brunoga/pipeliner/pull/264)). The previous "render both badges" fix ([#239](https://github.com/brunoga/pipeliner/pull/239)) made the JS emit both `search` and `list` badges into the DOM, but `.ve-chip { overflow:hidden }` (defined further up the stylesheet) silently clipped the trailing badge once the plugin name + role dot + both badges exceeded the palette's inner width. `bluray_releases` is the only plugin currently long enough to trigger it, but the failure mode was invisible — the badge existed in the DOM with non-zero size, just off-screen — so it looked like the prior fix had failed. The fix combines the palette-tightening pass above with `flex-shrink:0` on both badge classes (so any future plugin with an even longer name truncates the *name*, not the trailing badge — the capability indicator must remain visible) and an unrelated cleanup of the disabled-state palette render path that previously emitted chips with no badges at all (capability information is informational and should always show, whether or not the palette is interactable). The new `TestE2EBlurayPaletteChipShowsBothBadges` Playwright test compares badge bounding rects against the chip bounding rect — `IsVisible()` returns true even for overflow-clipped elements, which is exactly how the previous fix slipped past CI.

## [1.9.1] - 2026-06-08

### Changed

- **Password hashing moves from SHA-256 to bcrypt, with a failed-login throttle** ([#261](https://github.com/brunoga/pipeliner/pull/261)). The web UI stored credentials as raw SHA-256 hashes in memory. The hash never touches disk and the plaintext is already in argv (via `--web-password`), so the practical attack surface is narrow, but SHA-256 leaves brute-force unrate-limited if the hash ever leaks, and there was no per-attempt cost to slow remote guessing. `newCredentials` now uses `bcrypt.GenerateFromPassword` at default cost (~250 ms per comparison on modern hardware); `matches` always runs `bcrypt.CompareHashAndPassword` even when the username doesn't match, so login timing doesn't leak username existence. `handleLoginPost` adds a 300 ms `failedLoginDelay` sleep on every failure path as belt-and-suspenders in case a future code path short-circuits before bcrypt. A package-private `bcryptCost` variable defaults to `bcrypt.DefaultCost` in production but is dropped to `bcrypt.MinCost` by a test-only `init()` so the Playwright suite (~40 servers, each constructing credentials) doesn't blow past the package's CI test timeout. `SECURITY.md` is updated to reflect bcrypt and the failure-path throttle (was previously documented as "no rate limiting, SHA-256").
- **Config-reload lifecycle: new instances are built before old ones are torn down** ([#261](https://github.com/brunoga/pipeliner/pull/261)). `Server.reload` changes signature from `func() error` to `func(content []byte) error` and now owns the entire pipeline: parse → validate → `BuildTasks` → atomic write → commit the in-memory task swap. The old order was disk-write → reload-from-disk, with old `Shutdown()` happening before new construction; the new order builds the new instances first and only swaps if every plugin factory succeeds. A consequence visible to plugin authors: a plugin's `Shutdown()` may now be called on an instance that was constructed but never had `Generate` / `Process` / `Consume` / `Commit` invoked — when a sibling plugin's factory fails partway through a reload, the new instances that *were* built are cleaned up immediately and the old instances keep running. `PLUGIN_DEVELOPMENT_GUIDE.md`'s reload-lifecycle section is rewritten to describe the new order.

### Fixed

- **Visual editor zoom anchors on the cursor (wheel) and viewport centre (+/- buttons)** ([#260](https://github.com/brunoga/pipeliner/pull/260)). Mouse-wheel zoom previously updated `ve_zoom` without touching pan, so the world point at the canvas top-left stayed fixed and content slid toward the corner as the user scrolled in. The wheel listener was also attached to the inner scaled canvas, so wheel events over empty viewport space were ignored — only hovering over a node triggered zoom. Both bugs share a root cause: `setZoom` didn't anchor on anything. A new shared `zoomAt(z, cx, cy)` helper applies the same math the pinch-zoom path already used — `panX_new = cx - (cx - panX_old) * (zoom_new / zoom_old)` — and the wheel listener moves up to the `ve-canvas-body` viewport container so it catches wheel events anywhere. The `zoomIn` / `zoomOut` toolbar buttons had the same drift bug (different code path) and now anchor on the viewport centre via the same helper. `zoomReset` (100% button) now clears pan so the default-view button can't leave content scrolled off-screen; `zoomToFitHorizontal` clears horizontal pan for the same reason while preserving vertical pan to honour the user's scroll position. 13 new unit tests cover `zoomAt` invariants (cursor-fixed-point across zoom in/out, non-zero starting pan, min/max clamp no-ops), the button helpers (centre anchor, inverse round-trip, missing-body fallback), and the pan-clearing semantics of `zoomReset` and `zoomToFitHorizontal`.
- **Web UI session expiry no longer freezes the dashboard silently** ([#261](https://github.com/brunoga/pipeliner/pull/261)). `requireSession` redirected every unauthenticated request to `/login` via HTTP 303 — including XHR/fetch calls to `/api/*`. The browser auto-followed, the JS received a successful HTML response that didn't parse as JSON, and `dashboard.js`'s catch arm just set "error — retrying" forever with no way back to a usable UI. The user had to manually reload the page. `requireSession` now branches on `strings.HasPrefix(r.URL.Path, "/api/")`: API paths return `401` with a small JSON body, page paths still get the 303 redirect (browsers handle that one correctly). A global `fetch` wrapper installed in `index.html` (before any other JS loads) intercepts 401 responses to `/api/*` and navigates the browser to `/login`, so the SPA recovers from session expiry automatically. `TestRequireSessionBehaviorByPath` covers the 401-vs-303 split.
- **Config save is atomic and validates the full build before persisting** ([#261](https://github.com/brunoga/pipeliner/pull/261)). `apiSaveConfig` used to write the file via `os.WriteFile` before calling `reload`. Two problems: static validation only ran the descriptor-level `Validate` callbacks, not the plugin `Factory`s, so a config that was structurally valid but blew up at instantiation got persisted anyway — the response carried a warning string but the next process startup couldn't load. And `os.WriteFile` is not atomic, so a kill or disk-full mid-write could truncate the user's config to zero bytes. Together with the reload-lifecycle change above, the new flow is: parse → validate → `BuildTasks` → `config.AtomicWriteFile` (temp file in the same dir + rename) → commit the in-memory task swap. Any failure at any step leaves the on-disk file untouched and calls `Shutdown()` on the partially-built plugin instances. `apiReload` (POST `/api/reload`) reads disk first and feeds the bytes through the same path; `TaskDone` pulls queued bytes off `Server.pendingReloadBytes` when running tasks finish so the "save while busy" path also takes the new code path. `POST /api/config` also gains a 1 MB `MaxBytesReader` cap so a malicious LAN client can't buffer arbitrary memory by posting a huge JSON body. New `TestSaveConfigPreservesFileOnReloadError`, `TestSaveConfigRejectsOversizedBody`, and `TestAtomicWriteFile_*` (including a "leaves original on failure" case) exercise the new invariants.
- **Executor `mergeAndDedup` and `failedURLs` guard against empty-URL collisions** ([#261](https://github.com/brunoga/pipeliner/pull/261)). Both kept their map keyed on `entry.URL`. A future plugin emitting URL-less entries would have them all collapse onto the same `""` key — only the first would survive merge, and a single URL-less failure would suppress commit for every other URL-less entry. No current source plugin produces URL-less entries that reach the executor (the empty-URL `entry.New` calls live inside search and list helpers that bypass dispatch), but the sharp edge is cheap to remove: `mergeAndDedup` only tracks non-empty URLs in the seen map, and the executor skips empty URLs when populating `failedURLs` and short-circuits the lookup at commit time so they always commit. Two tests using a title-keyed source/sink exercise the URL-less paths directly.
- **Visual editor `deletePipeline` shifts label, region, and nodes by the same amount** ([#261](https://github.com/brunoga/pipeliner/pull/261)). `_labelY` and `_regionY` shifted unconditionally while each `n.y` was clamped to `≥ 0`. If a deleted pipeline was taller than the offset of a pipeline below it (only reachable through manual node dragging), the region top went negative while the nodes stayed at 0 — visually splitting the pipeline. All three coordinates now shift by the same `vacated` amount; the auto-layout and Tighten Pipelines tools can normalise any resulting negative values.

## [1.9.0] - 2026-06-06

### Added

- **3D badge in the Movies database tab** ([#257](https://github.com/brunoga/pipeliner/pull/257)). The movies tracker keys 3D and non-3D versions of the same title under distinct bucket entries — `recordKey()` mixes `Is3D` into the bucket key by design, so they're independent records — so a user who downloads both ends up with two rows for one title in the Movies view. There was no visual cue separating them: the rows looked identical and the duplicate was confusing. The Title cell now shows a small accent-coloured `3D` pill when `rec.is_3d === true`, styled to match the existing badge family (uppercase, 9px, accent border on transparent background).

### Changed

- **DAG field-certainty analysis is now per entry state** ([#251](https://github.com/brunoga/pipeliner/pull/251), [#253](https://github.com/brunoga/pipeliner/pull/253)). The validator's single `Certain` set carried an implicit assumption that rejected and failed entries don't flow downstream — every narrowing rule (`condition` accept/reject, `require`, route ports, semantic groups) promoted a field to certain on the basis that "entries lacking the field were rejected, so what flows past has it". The 1.8.0 `swap_state` plugin deliberately broke that invariant by reviving rejected/failed entries, and the validator silently accepted configs where the revived entries lacked fields the narrowing had promoted: `... → require(fields=["video_year"]) → swap_state(swap=["rejected","accepted"]) → consumer(Requires=video_year)` validated as OK even though the new Accepted entries provably lack `video_year` — at runtime the consumer would silently skip every entry, or `pathfmt` would render `{video_year}` as the empty string. The fix is structural. Each node now carries four state buckets — Undecided / Accepted / Rejected / Failed — recording what's certain on entries currently in each state, plus a `populated` bitmask tracking which states actually contain entries at this point in the graph. Source plugins populate Undecided; classifier-shaped processors (with non-empty `Produces` — `series`, `movies`, `premiere`, `dedup`) populate their `EffectiveInputStates` and inherit the upstream Undecided bucket into the freshly-populated Accepted bucket, because matching entries flow Undecided→Accepted carrying whatever certainty they already had. `condition` accept rules go through a new `promoteAccepted` primitive that copies Und into Acc and adds the promoted set (so a downstream sink reading Accepted-only correctly sees the field) — without this, accept rules never populated Accepted and the visual editor's "Fields available at input" preview was wrong for any Accepted-reading consumer. `condition` reject rules and `require` go through `narrowAcceptedUndecided`, which promotes fields on the passing buckets and intersects the Rejected bucket against the certainty of the newly-rejected entries (without the absence-checked field) — so a downstream `swap_state` that revives those rejected entries correctly defeats the narrowing. Route ports apply port-accept narrowing only to the passing buckets. `swap_state` exchanges two buckets and their populated bits. A downstream `Requires` check intersects the buckets named in the consuming plugin's `EffectiveInputStates`, with a fallback to `(Accepted ∪ Undecided) ∩ populated` when none of the requested states are populated upstream — so a structurally unreachable node (the typical "no Accept filter in this pipeline" case) doesn't emit noise warnings. The visual editor mirrors the entire model client-side in `internal/web/ui/visual-editor.js` so the live "Fields available at input" panel, condition-builder hints, and `fieldWarnings()` emit the same answers `pipeliner check` returns — without this the live preview was loose (it propagated every Produces and accept-promoted field as certain regardless of state) and silently disagreed with the server after a `swap_state`, `require`, or merge. `/api/plugins` exposes `input_states` (string[]) so the JS knows which buckets each consuming node reads. All 22 sample configs in `configs/*.star` validate identically to the prior model. New tests in `internal/dag/swap_state_test.go` (require+swap, condition+swap, benign-swap) and `internal/web/ui/tests/state-certainty.test.js` (the same scenarios mirrored client-side) keep the two sides in lockstep.
- **`swap_state` README and `PLUGIN_DEVELOPMENT_GUIDE.md` document the per-state model** ([#254](https://github.com/brunoga/pipeliner/pull/254)). The swap_state README's stale "the DAG validator's Reachable/Certain field analysis does not yet model what swap_state does to the entry population downstream" gotcha is replaced with a description of the actual per-state behaviour and the two patterns that avoid spurious downstream warnings (place the swap right before a no-`Requires` sink — the dedup-cleanup pattern — or add a downstream `require`/`condition` to re-establish the contract on the revived population). `PLUGIN_DEVELOPMENT_GUIDE.md` gains a new "DAG field certainty" subsection covering source vs classifier-shaped population, the five narrowing mechanisms (`condition` accept via `promoteAccepted`, `condition` reject + `require` via `narrowAcceptedUndecided`, route ports, semantic groups, `swap_state`), the `EffectiveInputStates` intersection check and its fallback for structurally unreachable nodes, and the visual editor's client-side mirror. The Requires descriptor-field row is updated to reflect per-state buckets (was previously framed as "in Produces of every upstream path"). The State-mutating plugins section gains a forward note that any new state-mutating plugin must teach the validator about its effect on the buckets in `internal/dag/validate.go`.

### Fixed

- **Visual editor: Copy/Cut toolbar buttons appear after direct-click selection** ([#255](https://github.com/brunoga/pipeliner/pull/255)). `selectNode()` (the direct-click path) set `ve.selectedNodeId` but never called `updateCopyButton()`, so single-click selection left Copy/Cut hidden. Region-select reached them indirectly through `updateExtractButton → updateCopyButton`, which is why the bug only surfaced for the direct-click case. The call now lives inside `renderParamPanel()`; every selection-change path already routes through there (selectNode, toggleMultiSelect, marqueeSelect, the Escape key handler, the empty-canvas-click deselect, drag-start, removeNode, function-editor swap) so the toolbar stays coherent with the panel without scattering `updateCopyButton()` calls across every mutation site. A new `TestE2ECopyButtonAppearsAfterDirectClick` Playwright test guards the regression.
- **Visual editor: pasted nodes land inside the active pipeline's region** ([#256](https://github.com/brunoga/pipeliner/pull/256)). `pasteClipboard()` placed new nodes at the viewport's vertical centre. When the user was working on a pipeline whose region sat below that centre (any second-or-later pipeline in a multi-pipeline config), the pasted node landed above the active region — visually disconnected — and `startNodeDrag`'s `effMinY = g._labelY + 36` floor snapped it downward into the region the instant the user grabbed it to move. That looked like the node "jumping" with no visible cause. The paste y is now clamped to `[g._labelY + 36, g._regionY + g._regionH - NODE_H - 24]` so the new node falls inside the active pipeline's vertical band; the viewport-centre is still honoured when it falls inside the band, so single-pipeline configs are unaffected. A new `TestE2EPasteLandsInsideActivePipelineRegion` Playwright test reproduces the bug with a two-pipeline config (pasted y=269 vs floor=300 in the reproducer) and passes with the clamp.
- **Condition README documents the bool-field comparison pitfall** ([#252](https://github.com/brunoga/pipeliner/pull/252)). Missing fields evaluate to `""`, not the typed zero value, so `bluray_3d_release == false` does not match entries where the upstream plugin never set the field — `compareEq` ends up doing `"" == "false"` (after string coercion), which is false, so the rule never fires. The recommended pattern is `!= true` against the truthy literal — that way both "set to false" and "not set at all" trigger the rule. The same trap applies to `video_is_3d`, `enriched`, and `video_proper`.

## [1.8.0] - 2026-06-05

### Added

- **`swap_state` processor swaps entries between two states** ([#244](https://github.com/brunoga/pipeliner/pull/244)). Motivating use case: a cleanup pipeline that should delete `dedup`'s rejected losers without touching the accepted winners cannot be expressed with the existing plugin set — sinks consume only Accepted entries, `dedup` rejects the losers, so no existing path delivers the losers to a delete sink while still excluding the winners. `swap_state(swap=["accepted", "rejected"])` does this in one node: the bidirectional flip turns losers Accepted (reaching the delete sink) and winners Rejected (filtered out by the sink), so a `filesystem → metainfo_file → dedup → swap_state → exec("rm {file_location}")` chain correctly targets only the duplicates. Other patterns the bidirectional swap unlocks: rescue failed entries through a fallback sink (`swap=["accepted", "failed"]`), re-feed rejected entries through a relaxed downstream filter (`swap=["rejected", "undecided"]`). `AcceptReason` / `RejectReason` / `FailReason` are preserved across the swap as audit history (so notify templates rendering `{{.FailReason}}` still work for rescued failed entries), and a new `Entry.LastStateChange` field — `{From, To, Plugin, Reason, At}` — records the transition itself so consumers can distinguish "always accepted" (`LastStateChange == nil`) from "rescued by a swap". The `Consumed` flag is orthogonal to State and is left alone across the swap; `Entry.Clone` deep-copies `LastStateChange` so per-fan-out clones don't share the pointer. Worked examples in the plugin README cover cleanup, retry/failover, and re-feeding through a relaxed filter; `configs/dedup-cleanup.star` ([#249](https://github.com/brunoga/pipeliner/pull/249)) is the canonical validating sample with a header that walks through *why* the bidirectional swap is what makes a single sink correctly target only the duplicates.
- **Declarative `Descriptor.InputStates` replaces the per-plugin "skip rejected/failed" convention** ([#245](https://github.com/brunoga/pipeliner/pull/245), [#246](https://github.com/brunoga/pipeliner/pull/246), [#247](https://github.com/brunoga/pipeliner/pull/247)). Every processor in the codebase used to open its `Process` method with `if e.IsRejected() || e.IsFailed() { continue }` (23 of 28 carried the canonical form; 3 had non-default hand-rolled variants). It was pure convention — no framework enforcement, no compile-time check, no lint rule — so a new plugin author who forgot the guard would silently operate on rejected entries. Plugins with broader or narrower state needs (`accept_all`, `dedup`, `limit`, the new `swap_state`) had to write a different bespoke guard. `Descriptor.InputStates` (typed `entry.StateSet`, a uint8 bitset over the four states) declares which states the plugin's `Process` or `Consume` method actually acts on; the executor pre-filters upstream entries to this set before calling the plugin, and merges the excluded entries back into the downstream slice so they continue to flow for bookkeeping and downstream visibility. Pre-built named constants cover every common case — `StatesAcceptedOnly`, `StatesUndecidedOnly`, `StatesAcceptedUndecided`, `StatesAllButFailed`, `StatesAll`. `EffectiveInputStates()` returns the declared value or a role-appropriate default (`StatesAcceptedUndecided` for processors, `StatesAcceptedOnly` for sinks); sinks additionally have an always-on consumed-exclusion at the boundary because `consumed` is orthogonal to State. The sweep retired the 23 canonical skip-guards in processors, added explicit declarations on the 3 exception plugins (`accept_all=StatesUndecidedOnly`, `dedup=StatesAcceptedOnly`, `limit=StatesAcceptedOnly`), retired the `entry.FilterAccepted(entries)` wrappers in all 9 production sinks, and removed the chained-sink special case in the executor (the per-sink pre-filter subsumes it). `swap_state` is the canonical example of a non-default declaration with `InputStates: entry.StatesAll`. Behaviour is unchanged for every existing plugin — defaults match what the per-plugin guards already did. The new `notify-on-failure` pattern (a chained sink with `InputStates: StateBit(Accepted) | StateBit(Failed)`) becomes expressible for the first time because `swap_state` cannot sit between sinks (the DAG validator only permits sink → sink chains); a user-facing Starlark config knob is a deferred follow-up but the underlying mechanism ships now.

### Changed

- **`pipeliner check` no longer warns on the FilterAccepted-style per-plugin gate** because that gate is gone — the executor's pre-filter does the same job declaratively. Validator warnings continue to surface stuff that matters: deprecated-field references, accept-only `condition` nodes, merge-gap field-certainty issues, and unsatisfied `Requires` groups.
- **Plugin development docs** ([#245](https://github.com/brunoga/pipeliner/pull/245), [#247](https://github.com/brunoga/pipeliner/pull/247)). `PLUGIN_DEVELOPMENT_GUIDE.md` gains an `InputStates` row in the descriptor-fields table, a new "Declaring InputStates" subsection with a per-set guidance table (when to reach for each named constant, with codebase examples), a new "Sinks specifically" subsection spelling out the always-on consumed exclusion and the notify-on-failure descriptor pattern, and updated `ProcessorPlugin` + `SinkPlugin` "Rules" sections (drop the obsolete "skip rejected/failed yourself" / "call FilterAccepted yourself" bullets; example snippets drop the redundant guards with a one-line comment explaining why they're absent). A new "State-mutating plugins" subsection under the Entry data model documents the contract for `swap_state`-style plugins: declare `InputStates: StatesAll`, populate `LastStateChange` on every transition, preserve `AcceptReason` / `RejectReason` / `FailReason` as audit history.

### Removed

- **Per-plugin skip-guards from 23 processors** ([#246](https://github.com/brunoga/pipeliner/pull/246)) and **`entry.FilterAccepted(entries)` wrappers from 9 sinks** ([#247](https://github.com/brunoga/pipeliner/pull/247)). Both classes of boilerplate are subsumed by the `InputStates` pre-filter and add no value above it. Four plugin-internal tests went away because they were testing the in-plugin guard behaviour — passing pre-rejected/failed entries directly to `Process` and asserting the entry was not mutated. With the new contract, `Process` is only called with entries in the plugin's declared `InputStates`; exercising it with out-of-contract input is non-meaningful. The behaviour they covered is now enforced centrally by the executor and tested in `internal/executor/executor_test.go`.
- **Executor chained-sink special case** ([#247](https://github.com/brunoga/pipeliner/pull/247)). The line `produced = entry.FilterAccepted(upstream)` that hard-coded the "chained sinks only see what the upstream sink accepted" rule is gone — the same effect now falls out of the next sink in the chain pre-filtering its own input via the general `InputStates` mechanism.
- **`entry.FilterAccepted` helper** is kept callable for backward compatibility and for use in tests as a one-line equivalent of "the entries a default sink would actually receive". Its docstring is updated to reflect that it's no longer called by production sinks or by the executor's chained-sink path.
- **Unused `entry.FieldTypeOf` helper** ([#248](https://github.com/brunoga/pipeliner/pull/248)). Leftover from an earlier metadata refactor; no production caller, no test caller, no docstring reference. `WithBaseURL` (test-only) and `RequireAny` (test-only, kept to preserve AND-of-OR Requires semantics for future plugins) remain — both are documented false-positives from `deadcode ./...`.

## [1.7.1] - 2026-06-05

### Added

- **`pipeliner check` warns when a `condition` node has no reject expression** ([#241](https://github.com/brunoga/pipeliner/pull/241), [#242](https://github.com/brunoga/pipeliner/pull/242)). A condition with only `accept` rules — no top-level `reject` in the single-rule form, no item with a `reject` key in the `rules` list — only Accepts matching entries; the non-matching ones stay `Undecided` and `PassThrough` forwards them as-is. The filter therefore silently does nothing for most entries, despite the natural reading of "keep only entries where X". The motivating case: a `movies-3d-discover` pipeline used `condition(accept="bluray_3d_release == 'true'")` after `metainfo_bluray` to keep only releases with a real 3D edition, but every non-3D entry sailed through to the print sink — `in=80 out=80` with zero rejects logged. The new warning fires from `dag.Validate`, names the node, and shows the two correct patterns (`reject="not (X)"` or `rules=[{"accept": X}, {"reject": "true"}]`); it flows through both `pipeliner check` and the web UI's `/api/config` validation endpoint without any plumbing changes since both already consume `warnings` from `dag.Validate`. A second related pitfall is documented alongside in the plugin README and user guide: comparing bool fields with `== false` doesn't match missing fields because the expression evaluator returns `""` for unset identifiers (so `"" == "false"` is false); the recommended pattern is `!= true` against the truthy literal, which correctly rejects both "set to false" and "not set at all". Three bundled sample configs (`bluray-discover.star`, `jackett-tv-transmission.star`, `route-port-contracts.star`) were silently exhibiting the accept-only bug and are rewritten with `reject` so they validate cleanly and now demonstrate the right pattern for anyone copying them.

### Fixed

- **`bluray_releases` strips trailing 3D / 4K format tokens from emitted Title so list= consumers fuzzy-match correctly** ([#238](https://github.com/brunoga/pipeliner/pull/238)). Calendar rows are titled e.g. `"Avatar 3D"` / `"Inception 4K"` on Blu-ray.com, and `entryFromCalendar` / `entriesFromIndex` were passing those tokens through to `Entry.Title` verbatim. When the plugin is used as a list source (`movies.list=["bluray_releases"]`), `ResolveDynamicList` reads `Entry.Title` and feeds it through `match.Normalize` — which lowercases and collapses separators but does *not* strip `3d` / `4k`. An incoming release parsed as `"Avatar"` therefore never fuzzy-matched the `"avatar 3d"` list entry and was silently rejected by the `movies` filter. The two emit sites now use `stripFormatToken(...)` for both `Entry.Title` and `FieldTitle`; format information remains structurally available via `bluray_format` and `bluray_is_3d_edition`, so nothing downstream loses information. The plugin's own `indexKey` already stripped the same tokens, so the cache layout is unchanged.
- **Visual editor renders both search and list badges on palette chips for plugins that advertise both capabilities** ([#239](https://github.com/brunoga/pipeliner/pull/239)). The palette chip render in `visual-editor.js` used a mutually-exclusive ternary — `is_search_plugin ? 'search' : is_list_plugin ? 'list' : ''` — so a plugin (or user function) registered with both `IsSearchPlugin: true` and `IsListPlugin: true` only ever showed the SEARCH badge. `bluray_releases` is the canonical case. The hover-accent CSS class and drag tooltip had the same bug. Both badges, both classes, and both tooltip lines now render side-by-side. Canvas sub-node rendering at L675-679 is unchanged — those badges are driven by how the node was wired in the graph (`isSearchNode` xor `isListNode`), not by the plugin's capability set, so the existing ternary is correct there.
- **`deluge` sink logs the offending URL on failure, tightens URL validation with `net/url.Parse`, and summarizes the cryptic Twisted `Unsupported scheme: b''` traceback into a one-line error** ([#240](https://github.com/brunoga/pipeliner/pull/240)). Two prior fixes ([#201](https://github.com/brunoga/pipeliner/pull/201), [#204](https://github.com/brunoga/pipeliner/pull/204)) tightened source-side URL validation in `rss` and `jackett` and added a `strings.HasPrefix` guard in the deluge sink, but the `Unsupported scheme: b''` failures kept appearing on `tvshows-favorites` runs against Jackett. The reason the third fix was hard was that the deluge error log only included the entry title — the URL bytes actually being passed to the daemon were invisible, so every fix was a guess. Three changes: (1) `validateTorrentURL` replaces the `HasPrefix` check with `net/url.Parse` and rejects URLs whose scheme is anything but `http`/`https`/`magnet`, and additionally requires a non-empty `Host` on http(s) URLs — catching the degenerate-but-prefix-passing shapes (`"http://"`, `"/dl/foo.torrent"`, `":foo"`) that surface in Twisted as the `b''` scheme. (2) The "deluge: add torrent" error log now carries `url` and `link_type` alongside the existing title + err, so the next occurrence finally captures what URL is failing. (3) `summarizeAddError` detects the wall-of-Python-traceback deluge returns for the empty-scheme case (and the named-scheme variant `b'magnet'` etc.) and rewrites it as a one-line message that names the URL and the likely cause (broken indexer redirect Location header). The entry's `FailReason` is no longer a multi-paragraph Python traceback.

## [1.7.0] - 2026-06-04

### Added

- **`bluray_releases` source and `metainfo_bluray` processor scrape Blu-ray.com for catalog-grounded release metadata** ([#234](https://github.com/brunoga/pipeliner/pull/234)). The motivating problem: `metainfo_file` sets `video_is_3d=true` for anything tagged "3D" in a release filename, but a large fraction of those are upscales, conversions, or mis-tagged half-rips rather than real BD3D source material. Without an external ground truth there was no way for a pipeline to tell a genuine BD3D rip apart from a fake. The new plugins introduce that ground truth from Blu-ray.com's release catalog. `bluray_releases` (source) scrapes the monthly release calendar and is also usable as a `SearchPlugin` inside `discover.search` and as a list source inside `series.list` / `movies.list`. `metainfo_bluray` (processor) enriches movie entries with the matching catalog record — release date, studio, country, year, runtime, codec (e.g. `MPEG-4 MVC` = the BD3D codec), resolution, aspect ratio, edition, format (`BD` / `UHD` / `BD3D` / `DVD`) — and the two new 3D booleans: `bluray_3d_release` is true when the *movie* has any 3D release in the catalog, and `bluray_is_3d_edition` is true when *this specific release record* is the 3D edition. The fake-3D filter is then `video_is_3d == true and bluray_3d_release != true`. Three SQLite-backed caches keep request volume low: `cache_bluray_index` (720h, normalised-title → `[]IndexEntry` map, populated by both calendar passes and search fallbacks so it accumulates across runs), `cache_bluray_detail` (168h, full `Release` per ID), and `cache_bluray_search_neg` (168h, "we searched, found nothing" markers that prevent re-hitting `/search/` for known misses). Lookup order is in-memory index → SQLite index → negative cache → `/search/` (write-through). A weekly `bluray_releases` source pass warms the index so per-entry enrichment usually never hits `/search/`. New field constants in `internal/entry/info.go`: `bluray_id`, `bluray_url`, `bluray_format`, `bluray_studio`, `bluray_country`, `bluray_year`, `bluray_release_date`, `bluray_runtime_minutes`, `bluray_codec`, `bluray_resolution`, `bluray_aspect_ratio`, `bluray_edition`, `bluray_3d_release`, `bluray_is_3d_edition`. Sample configs cover each role independently: `configs/bluray-calendar.star` (source), `configs/bluray-enrich.star` (processor + `route()` fake-3D detector), `configs/bluray-discover.star` (search backend inside `discover`). Tier 2/3 catalog data (HDR variants, audio/subtitle tracks, packaging photos, prices, ratings) is deliberately out of scope for v1.
- **`bluray_releases` auto-routes to the dedicated 3D calendar when `formats=["BD3D"]` and supports historical backfill** ([#235](https://github.com/brunoga/pipeliner/pull/235)). Blu-ray.com publishes `/3d/releasedates.php` with the exact same `movies[N] = {…}` JS-array structure as `/movies/releasedates.php` (so the same parser handles both) but server-side filtered to BD3D releases only. For BD3D-only configurations this is dramatically cheaper: a typical recent month has 100+ rows on the generic calendar with only 0-3 BD3D entries, the rest parsed and discarded by the client-side format filter. When the configured format set is exactly `{BD3D}`, `Generate` now hits `List3DMonth` instead — which also forces `Format=BD3D` on every parsed row, fixing edge-case titles like `"Alita: Battle Angel 4K and 3D"` that don't end with the ` 3D` token `FormatFromTitle` expects. Default and multi-format configs still hit `/movies/releasedates.php`, so existing pipelines are unaffected. The `windows()` cap is also lifted from 240 to 600 months (50 years) — it's a malformed-config guard, not a user limit, and 600 months covers the entire BD3D era (2010-09 → today) plus future headroom. With that change a one-shot historical backfill of the BD3D catalog is just `input("bluray_releases", from_year=2010, from_month=9, to_year=…, to_month=…, formats=["BD3D"])` — ~180 small monthly fetches at the default 1 req/sec, ~3 minutes total. After it runs the title index is fully warm and `metainfo_bluray` enrichment for any 3D-era movie hits `cache_bluray_index` without an HTTP round-trip. No new fields, config keys, or descriptors — pure behaviour change inside `Generate()`. Other 3D-related endpoints audited and rejected as not worth wiring: the `/3d/` landing page carries strictly less data than the calendar (no release date, studio, year, edition), `/movies/movies.php?format=3d` silently ignores the format filter and serves the generic catalog, `/search/?section=3d` returns zero rows without a keyword, and sibling `/4k/releasedates.php` / `/dvd/releasedates.php` 404.

### Fixed

- **`movies` tracker tolerates ±1 year release-year drift between theatrical and Blu-ray releases** ([#236](https://github.com/brunoga/pipeliner/pull/236)). The tracker key is `(title, year, is3D)`, but release filenames disagree on which year encodes a film — festival/theatrical year versus Blu-ray release year — depending on the encoder's convention. The motivating case was *Good Boy* (2025 theatrical / 2026 Blu-ray): a 2160p copy was downloaded and stored as `good boy|2026`, then two days later a 1080p H.264 rip with `2025` in the filename arrived through a different pipeline. The exact-key lookup for `good boy|2025` missed, `IsSeen` returned false, and the 1080p got accepted as a brand-new movie with reason `"matched"` instead of `"already downloaded"`. The same brittleness defeated quality-upgrade detection in the reverse direction: a stored 1080p at year=2026 plus a drifted 2160p at year=2025 would skip the upgrade check entirely (since `Latest(...).Year != year` short-circuited it). `IsSeen` now falls back to a year-tolerant scan (`LatestNearYear`) when the exact key misses and `year != 0`, and the `movies` filter calls `LatestNearYear` in place of `Latest + rec.Year == year` so the upgrade decision uses the same window. The tolerance is `±1` year — wide enough to absorb theatrical/home-video drift, narrow enough that genuinely independent films sharing a title stay distinct (e.g. *The Matrix* 1999 vs. a 2021 reboot, ≥2 years apart). The existing `year == 0` fallback (yearless filenames matching previously-enriched real years) is preserved. The fix is independent of the 1.5.0 quality-plugin refactor — the dedup path was unchanged across that refactor — but the year-key brittleness predates it.

## [1.5.0] - 2026-06-03

### Added

- **`limit` filter plugin caps how many accepted entries pass through, optionally taking the top N by a sort field** ([#226](https://github.com/brunoga/pipeliner/pull/226)). Motivated by manually-triggered "go wide" pipelines — e.g. a `discover` over a full Trakt watchlist that yields 70+ candidates the user wants to download a few at a time. With just `n=`, the first N accepted entries in arrival order win and the rest are rejected with `limit: N-entry cap reached`. With `sort=<field>` (and optional `order="asc"|"desc"`, default `desc`), entries are stable-sorted by that field and the top N win — the reject reason becomes `limit: N-entry cap reached (ordered by <field> <order>)` so the log explains *why* a particular release lost its slot. Sort values are typed: numeric (`int`, `int64`, `float64` cross-comparable), strings, `time.Time`, and booleans are all supported, and string-encoded dates from RSS (`published_date` in RFC822/RFC3339/etc.) are promoted to `time.Time` so they sort chronologically rather than lexicographically. Entries missing the sort field are bucketed last regardless of `order`, so a present-value entry always beats a missing-value one. The shared `internal/dateparse` package (RFC3339, RFC822, common feed layouts) is extracted from `age` and reused here, removing its private layouts list.
- **`quality` filter plugin gates entries by parsed `_quality` against a spec** ([#227](https://github.com/brunoga/pipeliner/pull/227)). Spec parsing used to live three times — once each inside `series`, `movies`, and `premiere`. The standalone filter replaces that duplication and turns quality into a normal composable DAG node. Required `spec=`, optional `on_missing` (`pass` default, or `reject` to drop entries without a `_quality`). The reject reason names both the parsed quality and the spec — `quality: 480p WEB-DL does not match spec "720p+"` — so the log makes the gate visible. Sample configs are rewritten to place `process("quality", spec=…)` upstream of the tracker plugins.
- **Visual editor marks auto-migrated nodes** ([#227](https://github.com/brunoga/pipeliner/pull/227)). When a config-load migration injects a node that doesn't exist in the text source, the new `dag.Node.AutoMigrated` field flows through to the editor's JSON response and the node renders with a dashed amber border and an `auto-migrated` badge whose tooltip names the migration. The marker bubbles up to function-call cards too — when the injection lands inside a `# pipeliner`-annotated function body, the parent call site shows the badge so users don't edit a node that doesn't exist on disk. The same migration logic runs inside the function-body editor (a JS-only parse path that bypasses the Go-side Starlark exec), and handles both literal values (`quality="720p+"`) and param references without defaults (`quality=quality`) — saving from the function-body editor persistently migrates the function source to the new form.

### Changed

- **`quality=` config key removed from `series`, `movies`, and `premiere` — quality gating moves to the new `quality` filter** ([#227](https://github.com/brunoga/pipeliner/pull/227)). The three tracker plugins now own *only* identity + tracker logic; spec matching is a separate upstream node. A config-load migration registry (`internal/config/migrations.go`) keeps every legacy config working: when Starlark execution sees `quality=` on one of the three plugins, the key is stripped from the call's config map and an explicit `process("quality", spec=<value>)` node is injected upstream, with a deprecation warning surfaced via `Config.LoadWarnings` (printed by `pipeliner check`). The registry is the surface for future deprecations — each entry is a self-contained `Migration{Name, Apply}` value and adding the next one is a small append. The text source is never rewritten; the in-memory graph carries the injected node tagged so the visual editor can highlight it. Users see the new form in samples and the user guide; existing configs keep running until the user updates them.
- **`series` and `movies` share a single PROPER/REPACK upgrade decision** ([#227](https://github.com/brunoga/pipeliner/pull/227)). Both plugins call `internal/quality.Decide(incoming, stored, properOrRepack, storedIsRepack)` which returns `UpgradeNo`, `UpgradeQuality`, or `UpgradeProperRepack`. The old per-plugin `if betterQuality || (properOrRepack && notDowngrade)` blocks are gone, and the two plugins now agree exactly on what "the same release reappearing in the feed shouldn't re-download forever" means.
- **Shared config-parsing helpers in `internal/plugin`** ([#227](https://github.com/brunoga/pipeliner/pull/227)). `IntVal(any, def) int`, `ToStringSlice(any) []string`, and `OptBool(cfg, key, def) bool` move out of the per-plugin copies in `series`, `movies`, and `premiere` and into `internal/plugin/helpers.go`. Other plugins still keep their own copies for now — out of scope for this refactor; they can migrate incrementally.

### Fixed

- **`series` no longer re-downloads the same PROPER/REPACK release on every run** ([#227](https://github.com/brunoga/pipeliner/pull/227)). `series.Record` had no `Repack` field, and the upgrade check was `betterQuality || (properOrRepack && notDowngrade)` — so the second time a `S01E01.REPACK.720p` release appeared in a feed (with the first already downloaded as the stored record), it satisfied the `notDowngrade` branch and got accepted again. And again. The fix mirrors what `movies` already did: `series.Record` gains a `Repack bool`, populated at commit time from `video_proper`/`video_repack`, and the shared `quality.Decide` adds `!storedIsRepack` to the PROPER/REPACK branch so chaining REPACK → REPACK at the same quality is blocked. Pre-existing records (written before this field existed) decode as `Repack: false`, costing one extra re-download in the worst case before the new record is stamped with the correct flag.
- **Visual editor keeps sub-nodes inside the pipeline region during drag** ([#228](https://github.com/brunoga/pipeliner/pull/228)). Dragging a parent node with an attached `list` or `search` sub-node (or a `route` parent with port children) could push the sub-node outside the pipeline region — most visibly, a `list` card sitting above a `movies` parent ending up above the pipeline label when the parent hit the top boundary. Only the parent's `minDragX` / `minDragY` was being clamped; sub-nodes inherited the same translation delta and slipped through. Both drag paths (single-node and multi-node) now compute the parent's *effective* clamp at drag start by walking the snapshotted sub-node offsets — the parent must stay far enough below/right that every sub also clears the region edge. Layout-agnostic: only subs above or to the left of their parent add a constraint (list plugins usually do); subs below or to the right add none (search plugins usually do) because the region grows downward/rightward to fit them.

## [1.4.0] - 2026-06-02

### Added

- **`trailer` filter plugin drops trailer / teaser / featurette / behind-the-scenes entries (or keeps only them)** ([#222](https://github.com/brunoga/pipeliner/pull/222)). Trackers and discovery feeds occasionally publish trailers alongside the full release — the 3D-movies discover log that motivated #221 surfaced `Avatar: Fire and Ash (2025) IMAX 3D Full SBS Trailer 1` inside what was meant to be a full-movie pipeline, and there was no built-in way to filter it out short of a hand-written `regexp` rule. The new processor matches a small set of short-form-release tokens against the entry title with case-insensitive word boundaries — `trailer`, `teaser`, `sneak peek`, `featurette`, `behind the scenes`, `BTS` — and accepts space, dot, dash, or underscore as token separators so scene-style names like `Sneak.Peek` and `Behind_The_Scenes` still match. A single `mode` config key controls behaviour: `mode="reject"` (default) drops trailer-like entries, while `mode="accept"` rejects everything that is *not* a trailer (suitable for trailer-only feeds). Word boundaries keep movie titles such as "The Greatest Showman" or "Movie Featuring Star" from triggering false positives — `\bfeaturette\b` doesn't match `Featuring`.

### Fixed

- **`quality="3dfull"` no longer accepts half rips, conversions, AI upscales, or plain SBS / OU** ([#221](https://github.com/brunoga/pipeliner/pull/221)). The 3D format parser had three blind spots that all let releases past a `quality="3dfull"` gate which should have been rejected. (1) The compound-marker regex `re3D` only accepted dash-joined half markers — `H-OU`, `Half-SBS` — so the very common space-separated scene forms `H OU` / `Half SBS` matched only the trailing bare `SBS` / `OU` alternative and fell through to the Full case (the title that surfaced the bug was `Ballerina 3D 2016 1080p H OU BluRay x264 DTS 5 1 vice`). (2) Bare `SBS` and `OU` were classified as Full, but scene convention reserves the prefix-less form for half-resolution rips — a "1080p SBS" release cannot physically be full-SBS (which would be 3840×1080); full-resolution rips are explicitly tagged `F` / `Full`. (3) `re3DConv` recognised only `3D-Conv` / `3DConv` and the standalone word `CONVERT`, so the matching space form `3D Conv` (e.g. `Ballerina 2025 3D Conv FSBS DDP 5 1 Atmos AETHER`), bare `Conv` / `Conversion` / `HT Conversion`, `AI Enhanced` / `Upscaled`, and the AI conversion-tool signatures `StereoCrafter` / `DepthCrafter` were all unrecognised. `re3DConv` is now split into a *strong* group (`3D-Conv`, `HT Conversion`, `StereoCrafter`, `DepthCrafter`) that always implies Conv, and a *weak* group (bare `Conv` / `Convert` / `Conversion` / `AI Enhanced` / `AI Upscaled` / `Upscaled`) that only implies Conv when a native 3D marker is also present in the same title — so a non-3D 4K-upscaled SDR remaster with `Upscaled` in the title isn't misclassified as 3D-Conv. `re3D` tolerates space separators in every compound marker (`Full SBS`, `F OU`, `Half SBS`, `H OU`, etc.) and `Parse` normalisation strips spaces too. `parseFormat3D` is updated symmetrically on the spec side (`sbs` / `ou` / `hsbs` / `hou` → Half, `fsbs` / `fou` / `full-sbs` / `full-ou` added as Full, `mvc` accepted as a BD3D spec token). **Breaking change for user specs:** configs that wrote `quality="sbs"` or `quality="ou"` to mean "full SBS / OU" now mean Half. Update them to `quality="fsbs"` / `quality="fou"` if you want full-resolution.

## [1.3.2] - 2026-06-02

### Fixed

- **`discover` caches results within the interval instead of skipping the title entirely** ([#216](https://github.com/brunoga/pipeliner/pull/216)). The per-title cooldown introduced in 1.3.0 was implemented as "skip the title and emit nothing until the TTL expires," so a 24h interval meant ~23h of every day the downstream pipeline saw zero entries for any cooldown-throttled title. The intent was to throttle the indexer re-search, not the downstream flow. The bucket schema gains a `results` field; within the cooldown the cached entries are re-emitted, and after the cooldown a fresh search refills the cache. Legacy pre-1.3.x records (and dry-run-poisoned records from before this fix) decode as `Results == nil` and are treated as a cache miss so they re-search immediately rather than producing empty output until the cooldown elapses. A legitimate "search returned zero results" cache is distinguished from a legacy record by initializing fresh result slices as `[]*entry.Entry{}` (drops `omitempty` so it encodes as `"results":[]` and decodes as a non-nil empty slice). Dry-runs now bypass the cache entirely — read bypass means a dry-run actually hits the search backends so "is jackett returning anything?" debugging works, and write bypass means a dry-run cannot stamp every title as just-searched and silently no-op the next real run within the TTL (the same idempotency principle as the executor commit phase in #209).
- **`Entry.Get` falls back to the matching struct field for `title` / `url` / `original_url` / `task`** ([#217](https://github.com/brunoga/pipeliner/pull/217)). `require(fields=["url"])` silently rejected every entry it saw with `missing required field: url`, even ones with perfectly valid `Entry.URL` values — and the same trap existed for `title`, `original_url`, and `task`. `require` only inspected the `Fields` metadata bag, and nothing in the codebase populates those four well-known names there; they live on the `Entry` struct. Templates and condition expressions already did the right thing because `interp.EntryData` mirrors the four struct fields into the lookup map before merging `Fields` on top, but `require`, the typed `Get*` helpers, and any other name-based consumer of `Fields` did not. `Entry.Get` (and the `GetString` / `GetInt` / `GetBool` / `GetTime` helpers that now delegate to it) now falls back to the struct field when the key isn't in `Fields`, for those four data-bearing names only. `Fields` takes precedence so metainfo plugins that rewrite `Fields["title"]` with a canonical TVDB/TMDb name continue to win over the raw struct title. Pipeline-machinery fields (`State`, `AcceptReason`, `RejectReason`, `FailReason`, `consumed`) stay out of scope.
- **Pipeline source order is preserved across dashboard, visual editor, and text editor** ([#219](https://github.com/brunoga/pipeliner/pull/219)). The text editor showed pipelines in source order (it's literally the file), but the dashboard and visual editor showed them alphabetically: `BuildTasks` iterated `sortedStringKeys(c.Graphs)` so the scheduler/dashboard order was always sorted, and `/api/config/parse` serialized `c.Graphs` (a Go `map`) whose `json` encoder sorts map keys, so the visual editor's `Object.entries(data.graphs)` always returned alphabetical order. The Starlark loader now records each `pipeline("name", …)` call into a `GraphOrder` slice; a repeated `pipeline()` with the same name overwrites the existing graph but keeps the name at its first-seen position so re-declarations don't reshuffle the UI. `BuildTasks` iterates `GraphOrder` (falling back to alphabetical for programmatically constructed configs), and `/api/config/parse` returns `graph_order` alongside the existing `graphs` map for the visual editor to consume.
- **Log-history scrollback no longer duplicates the visible tail or renders archived lines all-white** ([#215](https://github.com/brunoga/pipeliner/pull/215)). Two compounding bugs in the dashboard live-log scrollback from #199 made it look like the log wrapped around when scrolled to the top. First, `/api/logs/history` counts from EOF, so the first chunk a freshly-connected client requests is always the file's tail — exactly the slice the SSE warm-up just showed; the dashboard prepended it unconditionally, duplicating the visible tail. Second, the rotating log writer strips ANSI so on-disk logs stay grep-friendly, but `renderLogLine` only ran `ansiToHtml`, so the file-sourced duplicates (and genuinely-older lines from rotated archives) rendered all-white. `trimHistoryOverlap` now walks both the chunk and the page from their newest end backwards, dropping every matching pair (comparison strips ANSI so SSE-colored lines match their plain file copies); the server-side cursor still advances by the *original* chunk length so subsequent fetches read strictly-older lines. `plainLogToHtml` mirrors `clog/handler.go`'s `renderMsg` + level-color scheme so genuinely-older lines keep the same timestamp-gray + level-color + keyword-highlight look as the live tail; `renderLogLine` dispatches by ESC-byte presence so ANSI-bearing live lines still go through `ansiToHtml` unchanged.
- **Dry-run badge stays on the "Last run" row** ([#218](https://github.com/brunoga/pipeliner/pull/218)). After a dry-run the **DRY** badge was being rendered inside the right-side `<b>` of the "Last run" row, after the relative time and duration (`1m ago · 1.5s DRY`). On narrow task cards the combined string wrapped to a second line, which pushed the stats grid down and made dry-run cards visibly taller than their non-dry siblings. The badge is now anchored to the **"Last run"** label on the left side of the flex row — the right side stays compact, there is no wrap point in the timing string, and the badge ends up next to the label that describes what kind of run it was. `white-space: nowrap` on `.task-timing > span` acts as a safety net so future label/value growth can't reintroduce the wrap.

## [1.3.1] - 2026-06-02

### Fixed

- **`jackett` probes `t=caps` and falls back to `t=search` on Torznab error 201** ([#213](https://github.com/brunoga/pipeliner/pull/213)). After 1.3.0's #208, `jackett` began sending typed Torznab queries (`t=movie` / `t=tvsearch`) whenever the upstream entry had `media_type` set. Indexers that advertise only the generic `t=search` mode — 3dtorrents is the canonical example — reject those with Torznab error 201, so every search through such an indexer returned zero entries and a logged error. Each indexer's `t=caps` response is now fetched once on first use and cached for the lifetime of the process; the chosen search mode and typed params (`year`, `imdbid`, `tmdbid`, `tvdbid`, `season`, `ep`) are filtered against the caps' `available=` and `supportedParams=` lists, so 3dtorrents et al. get a plain `q=` query instead. If an indexer's caps are wrong or unparseable, a single fallback retry to `t=search&q=…` on a 201 response covers the gap. Caps fetch failures are deliberately *not* cached so a transient probe error doesn't permanently disable typed queries for an indexer.

## [1.3.0] - 2026-06-02

### Added

- **Dry-run mode now actually idempotent + UI controls** ([#209](https://github.com/brunoga/pipeliner/pull/209)). The dry-run docstring promised "fully idempotent" runs, but the implementation only skipped sink side effects — the commit phase still fired, so `movies` / `series` / `seen` / `premiere` trackers advanced their state and the *next* real run silently skipped the entries the dry-run had already "processed". The executor now skips the commit phase entirely when `ex.dryRun` is true, so dry-run is finally repeatable for inspection. The dashboard gets per-task **Dry** buttons and a global **Run all (dry)** action that fire dry-run triggers through the existing endpoints (which now accept `{"dry_run": true}` or `?dry_run=true`); runs surfaced as dry-run get a **DRY** badge on the last-run line.
- **`SearchPlugin.Search` receives the query entry, not just a title string** ([#208](https://github.com/brunoga/pipeliner/pull/208)). The string-only signature was a leftover from when `discover` only had a static title list; in a DAG world the upstream entry carries rich metadata that was being thrown away before the backend ever saw it. `Search(ctx, tc, e *entry.Entry)` now hands the full entry to backends, and `discover` forwards the upstream entry (preferring hint-rich entries over bare static titles when both share a title). In source mode, `Generate()` wraps the configured static query in a synthetic title-only entry so backends fall back to a plain title search.
  - **Jackett** translates available hints into typed Torznab params: `media_type` → `t=movie`/`t=tvsearch`/`t=search`, `video_year` → `year=`, `video_imdb_id` / `jackett_imdb_id` / `trakt_imdb_id` → `imdbid=` (canonical `tt` prefix stripped per Torznab convention), `jackett_tmdb_id` / `trakt_tmdb_id` → `tmdbid=`, `jackett_tvdb_id` / `tvdb_id` → `tvdbid=` (tvsearch only), `series_season` → `season=`, `series_episode` → `ep=` (tvsearch only). Server-side filtering is both more accurate ("It" 1990 vs 2017) and lighter on bandwidth than fetching N results and filtering downstream.
  - **RSS** exposes the same hints as new `url_template` variables (`{Year}`, `{IMDbID}`, `{TMDbID}`, `{TVDbID}`, `{Season}`, `{Episode}`, `{MediaType}`) alongside the existing `{Query}` / `{QueryEscaped}`. Integer hints render as empty string when zero so optional segments like `&year={Year}` don't emit `&year=0` in source mode.

### Changed

- **`jackett` emits `jackett_imdb_id` instead of `video_imdb_id`** ([#205](https://github.com/brunoga/pipeliner/pull/205)). `video_imdb_id` is the canonical IMDb ID field produced by metainfo enrichment plugins. Jackett was setting it directly from the Torznab `imdbid` attribute, while sibling indexer-provided IDs (`jackett_tvdb_id`, `jackett_tmdb_id`) followed the documented `jackett_`-prefixed hint pattern — `jackett` was the only source claiming a "canonical" `video_` field for unverified indexer data. The rename keeps `jackett` honest about what it produces (an indexer claim, not an authoritative ID) and lets downstream metainfo plugins promote hints to canonical fields uniformly. Configs that reference `video_imdb_id` immediately after a `jackett` source must rename it to `jackett_imdb_id`.

### Removed

- **`discover` no longer accepts a `list=` port** ([#206](https://github.com/brunoga/pipeliner/pull/206)). The `from=` / `list=` config key was a vestigial second path that did exactly what an upstream DAG connection already does, with worse error reporting (`Generate()` errors were silently dropped). The linear engine that originally required it has long since been removed, no sample configs use it, so the port, the `AcceptsList` flag on `discover`'s descriptor, and the backing parsing in `newPlugin` / `Process` are now gone. `series.list=` and `movies.list=` are untouched — those have a different filter-against-this-title-set semantic that upstream connections don't naturally express.

### Fixed

- **Visual editor preserves zoom and pan across text → visual switches** ([#210](https://github.com/brunoga/pipeliner/pull/210)). `switchView('visual')` unconditionally called `textToVisualSync()`, which resets `ve_panX` / `ve_panY` and runs `zoomToFitHorizontal()` — so flipping to the text editor and back wiped the visual editor's viewport. Visual ↔ dashboard was unaffected because it uses a separate top-level tab handler that doesn't touch canvas state. A `ve_textDirty` flag now tracks edits to the textarea (set by `oninput`, cleared at the end of a successful `textToVisualSync`); the re-sync is gated on the flag, so a clean tab flip preserves zoom and pan while an edited round-trip still re-fits as before.
- **`metainfo_tvdb` now actually sets `video_year` on enriched series** ([#207](https://github.com/brunoga/pipeliner/pull/207)). `metainfo_tvdb` declared `video_year` in `Produces` but never populated it — `SeriesInfo.Year` was always 0 and `SetSeriesInfo` only writes the field when `Year > 0`. Downstream nodes that required `video_year` after a `tvdb` step were treated as "certain" by the DAG validator but the field was missing at runtime. `Year` is now derived from the parsed `FirstAirDate` that TVDB already returns for every enriched series, matching the field's documented "premiere year" semantics.
- **`jackett` drops items whose link, enclosure, or GUID is not a fetchable URL** ([#204](https://github.com/brunoga/pipeliner/pull/204)). The previous guard only required `<guid>` to be a fetchable URL; an empty, whitespace-only, or scheme-less `<enclosure url>` or `<link>` still produced an entry that then surfaced downstream as a sink failure on an unusable URL. The parser now picks the first candidate (enclosure, link, GUID) that is a fetchable URL after whitespace trimming, and skips the item otherwise. Skipped items emit a warning log identifying the indexer and title so operators can see why a feed item went missing instead of it silently disappearing.

## [1.2.1] - 2026-06-01

### Fixed

- **Deluge sink rejects empty / scheme-less URLs locally** ([#201](https://github.com/brunoga/pipeliner/pull/201)). Entries arriving at the `deluge` sink with `e.URL == ""` (or a URL lacking `http://` / `https://` / `magnet:`) surfaced as a cryptic Twisted traceback from `core.add_torrent_url` — `twisted.web.error.SchemeNotSupported: Unsupported scheme: b''` — buried inside the deluge RPC error. The sink now fails fast with `entry has empty URL` or `unsupported URL scheme: "..."` before any RPC is attempted, so the actionable error lands on the entry's `FailReason` instead of in a Python stack trace.
- **`rss` and `jackett` no longer propagate non-URL feed identifiers** ([#201](https://github.com/brunoga/pipeliner/pull/201)). Both parsers fell back to the RSS `<guid>` (or Atom `<id>`) as `e.URL` when no link / enclosure was present. But RSS GUIDs with `isPermaLink="false"` are opaque identifiers (hashes, internal IDs), and Atom `<id>` is an IRI commonly of the form `urn:uuid:…` or `tag:host,date:…` — none of which are fetchable. The GUID / `<id>` fallback is now only accepted when the value actually starts with `http://`, `https://`, or `magnet:`; otherwise the item is dropped, matching the existing behaviour for items with no link at all. This is the root cause for the Deluge failures above.
- **Drag in the visual editor follows attached search / list sub-nodes** ([#202](https://github.com/brunoga/pipeliner/pull/202)). When a parent node owned a search / list sub-node (e.g. `discover` + `rss_search`, `series` + `tvdb_favorites`) or route ports, dragging the parent left the sub-nodes frozen in place — both for plain single-node drags and for multi-selections where the sub-node was visibly highlighted as part of the group. Stale sub-node positions then persisted into the `# pipeliner:pos` comment on save. Both drag paths now translate each attached sub-node by the parent's *post-clamp* effective delta, so they stay anchored to the parent even when the group hits a pipeline region boundary.

### Added

Replaces the generic `▶` glyph (which read as "play/media") with a small directed-graph mark that mirrors the visual editor's role palette: green source → blue processor → orange sink, joined by two short edges into a directed-arrow shape. The same SVG is served as the browser-tab favicon and used in the dashboard header, the login page, and the user-guide sidebar so the tab and the app read as one product. Served from the unauthenticated route so the login page renders the icon too.

### Fixed

- **`exitFunctionEditor` no longer drifts pipelines down on save** ([#197](https://github.com/brunoga/pipeliner/pull/197)). After editing a function, clicking *Save* inside the function editor, then *Save & Reload* on the config, every pipeline shifted down by ~one region-height — and *Save & Reload* persisted the drifted layout. `openFunctionEditor` stashes the live `ve.graphs` by reference (absolute y, `_regionY` set), and `exitFunctionEditor` called `initLayout()` directly on that restored reference; `initLayout`'s stored-position branch then added `_regionY` *again*. The conversion is now factored into a shared `initLayoutFromAbsolute()` helper used by both `exitFunctionEditor` and `undo()` (the latter already did the right thing manually), so the next call site that needs to restore absolute-coord graphs can't silently reintroduce the drift.
- **Marquee-select highlights owned `list` / `search` sub-nodes** ([#196](https://github.com/brunoga/pipeliner/pull/196)). Ctrl+Click on a parent that owns a list sub-node (e.g. `series` + `tvdb_favorites`) or a search sub-node (e.g. `discover` + `rss_search`) already highlighted both. Marquee-drag was leaving the sub-node un-highlighted until a full re-render fired, even though the parent ended up in the selection. `marqueeSelect` now mirrors the Ctrl+Click path: after adding a main node it walks `searchNodeIds` / `listNodeIds` and toggles the `multi-selected` class on each. Sub-node IDs themselves stay out of `ve.selectedNodeIds` — they ride along with their parent both visually and during "Extract to Function", matching the existing model.
- **Marquee-select activates the pipeline under the drag** ([#195](https://github.com/brunoga/pipeliner/pull/195)). Rubber-banding inside a pipeline that wasn't the active one silently selected nothing — the commit step only scanned `ve.graphs[ve.activeGraph]`. The marquee now picks the graph at the rectangle's centre via `findGraphAtPosition`, activates it through `setActiveGraph`, and only then iterates that graph's nodes. Marquees that don't intersect any pipeline region (the gap between two pipelines, or below all of them) are now a no-op rather than accidentally selecting nodes from whichever pipeline happened to be active.

## [1.1.1] - 2026-05-31

### Fixed

- **Title matching no longer tolerates single-character edits** ([#193](https://github.com/brunoga/pipeliner/pull/193)). `match.Fuzzy` previously accepted any pair within Levenshtein distance 1, which let "Masters of the Universe (2026)" match the unrelated movie "Master of the Universe (2026)" — and similar single-letter title flips elsewhere — causing the wrong content to download silently. Warnings flow through `pipeliner check` and the visual editor's inline validation. The condition/route field picker segregates deprecated names into a dedicated optgroup and renders chips with strikethrough.

- **`dedup` now requires `media_type` to be reachable upstream** ([#188](https://github.com/brunoga/pipeliner/pull/188)). The classifier signal is the trigger for both series and movie modes (formerly: `series_episode_id` alone for series, the now-deprecated `movie_title` for movies). For realistic pipelines this is invisible — `metainfo_file`, `metainfo_tmdb`, `metainfo_tvdb`, and the new classifier-filter `Produces` all set `media_type`. Pipelines that ran `dedup` without any metainfo upstream now get a clear validator error pointing at `media_type` instead of silently doing nothing useful.
- **All source plugins now populate `Fields["title"]`** with their raw entry name ([#184](https://github.com/brunoga/pipeliner/pull/184)). Previously `jackett`, `trakt_list`, `tvdb_favorites`, and `html` only set the `e.Title` struct field, so a bare `input → print` pipeline rendered an empty title. The tier model is documented in the user guide: sources fill `title` with raw → `metainfo_file` cleans it → external providers replace it with the canonical name.

Every plugin that wrote `movie_title` also wrote the same value to `title`, so the field is redundant. The `.torrent` display name (essentially the raw release filename) was overwriting the cleaned title set by `metainfo_file` or the canonical title set by `metainfo_tmdb` / `metainfo_tvdb` / `metainfo_trakt`.
